Introduzione

I servizi REST per il Polo digitale sono servizi di interfacciamento con altri sistemi e di trasformazioni tra i vari formati della piattaforma.

Risorse

In seguito vengono descritti i vari servizi esposti.

Sbnmarc

Una richiesta GET per ottenere il Sbnmarc di una risorsa.

Parameter Description

biblioteca

L’id della biblioteca per la parte di inventari e possessori (come SP,FC)

bid

Il bid del documento da cercare

Per esempio

$ curl 'http://localhost:8080/sbnmarc/search?biblioteca=SP&bid=ANA0000403' -i -H 'Accept: application/xml'

La riposta

HTTP/1.1 200 OK
Content-Type: application/xml;charset=UTF-8
Content-Length: 69

<?xml version="1.0" encoding="UTF-8"?>
<error>Read timed out</error>

Iccd

Una richiesta GET per ottenere l’Iccd di una risorsa. Questo metodo applica una trasformazione del record in format sbnmarc in ICCD, nel fomrato relativo al tipo di scheda. Al momento sopportato: F, S, D, e AUT per gli autori. AU.AUT ha la proprietÃ  'vid' che individuare la risorse collgata, authority file con vid, che puÃ² essere richiesto a questo servizio con bid=vid e type=AUT.

Parameter Description

biblioteca

L’id della biblioteca per la parte di inventari e possessori (come SP,FC)

bid

Il bid del documento da cercare

type

Il tipo della scheda F, S, D o AUT o null (senza)

version

La versione della scheda, per ora implementato solo per il tipo AUT o null, default=3

Per esempio

$ curl 'http://localhost:8080/sbnmarc/iccd?biblioteca=SP&bid=NAP0668034&type=F&version=4' -i -H 'Accept: application/json'

La riposta

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Content-Length: 72

{
  "status" : "ko",
  "message" : "Read timed out",
  "records" : [ ]
}

Per esempio per la scheda AUT

$ curl 'http://localhost:8080/sbnmarc/iccd?bid=SBLV224544&type=AUT&version=4' -i -H 'Accept: application/json'

La riposta per la scheda AUT

Content-Type: application/json;charset=UTF-8

Job Importazione

Una richiesta GET per importare il file Unimarc in formato JSON.

Parameter Description

filename

Percorso assoluto del file Unimarc

directory

La cartella di putput

profile

profilo au o na

Il processo di importazione Ã¨ diviso in bibliografico e authorities. Ci sarÃ  da lanciare:

  1. Import UNIMARC → Json

    Chiamata a: http://localhost:8080/jobs/import/start?filename=test.mrc&direcory=/opt/dataOut/etc

  2. Importazione in FE

    Chiamata a: http://localhost:8080/jobs/import/start?filename=test.mrc&direcory=toSolr

  3. Importazione in FE-Metaindice

    Chiamata a: http://localhost:8080/jobs/import/start?filename=test.mrc&direcory=toSolrMetaindice

Per Authorities Ã¨ sequenza Ã¨:

  1. Import UNIMARC → Json (AU)

    Chiamata a: http://localhost:8080/jobs/import/start?filename=test.mrc&direcory=/opt/dataOut/etc&profile=au

  2. Importazione in FE-Au

    Chiamata a: http://localhost:8080/jobs/import/start?filename=test.mrc&direcory=toSolr&profile=au

  3. Importazione in FE-Metaindice-Au

    Chiamata a: http://localhost:8080/jobs/import/start?filename=test.mrc&direcory=toSolrMetaindiceAu&profile=au
L’opzione aggiuntivo "clear=false" permette di fare un aggiornamento incrementale. Per default il servizio cancella tutti i record che sono stati inseriti prima dell’esecuzione del job di importazione.

Per esempio

$ curl 'http://localhost:8080/jobs/import/start?profile=naxml&filename=/opt/dataIn/___IE001_NAP_BN_00019690.mrc&directory=/opt/dataOut/output-20160223' -i -H 'Accept: application/json'

{
  "message" : "/opt/dataIn/___IE001_NAP_BN_00019690.mrc->/opt/dataOut/output-20160223",

Ricerca nel catalogo

GET

Una richiesta GET per cercare nel catalogo (opac, iccd, iccd-f, iccd-s, iccd-d, iccd-oa). NOTA: superato dalla richiesta in POST.

Parameter Description

q

query solr da fare, p.e. "roma"

fq

query per filtrare con le faccette, p.e. livello_bibliografico_s:monografia

Per esempio

$ curl 'http://localhost:8080/opac/search?q=roma&fq=livello_bibliografico_s:monografia' -i -H 'Accept: application/json'

POST

Una richiesta POST per cercare nel OPAC. Si manda in POST un oggetto del tipo SearchRequest creando una specie di protocollo uniforme per tutte le richieste anche piÃ¹ complesse.

Gli operatori sono;

  1. OR

  2. AND

  3. contains all [dei valori]

  4. contains one [dei valori]

  5. NOT

  6. between [valore1 e valore2]

  7. =

  8. <

  9. >

Ogni SimpleClause (proposizione semplice) ha un operatore esterno e uno interno. L’esterno serve in sostanza per la negazione della proposizione. L’interno della SimpleClause serve per dire come sono collegati i valori al campo, uguale o altro. Per il tipo CompostedClause l’operatore interno definisce la congiunzione tra le sue proposizioni (p.e. una lista di SimpleClause) in AND o in OR.

Se viene utilizzato l’operatore NOT (come operatore esterno) e l’interno Ã¨ "contains all" vuol dire che si esclude dal risultato solo i documenti che contengono tutte le parole. Con l’operatore "contains one" si escludono tutti i documenti che hanno una delle palore.
